# InProtect Client Bridge

These files are generated by launcher on each client start.

## What launcher writes

- `inprotect-session.json`
- `inprotect-session.sig`

Location: near your launched client executable/JAR.

## What data you get

- `user.uid`
- `user.displayName`
- `user.email`
- `user.role`
- `hasSubscription`
- `subscriptionUntil`
- `issuedAt` / `expiresAt`

## Runtime env vars passed to client process

- `INPROTECT_SESSION_PATH`
- `INPROTECT_SESSION_SIG_PATH`
- `INPROTECT_BRIDGE_SECRET`
- `INPROTECT_UID`
- `INPROTECT_NICK`
- `INPROTECT_EMAIL`
- `INPROTECT_ROLE`
- `INPROTECT_HAS_SUB`
- `INPROTECT_SUB_UNTIL`

## Minimal integration flow

1. On app start, read `INPROTECT_SESSION_PATH` and `INPROTECT_SESSION_SIG_PATH`.
2. Validate HMAC signature (`session.sig`) using `INPROTECT_BRIDGE_SECRET`.
3. Parse JSON and verify `schema`, `nonce`, `issuedAt`, `expiresAt`.
4. Ensure `expiresAt` is not in the past and token TTL is short.
5. Show player card from `uid`, `displayName`, `email`.
6. If signature/session is invalid, block protected modules.

Use `inprotect-session-reader.js` as an example helper.

## Important security notes

- No client-side protection is 100% uncrackable. Goal is to increase crack cost and detect tampering.
- Always distribute obfuscated release JARs (ProGuard/R8) and keep mapping files private.
- Keep SHA-256 integrity checks enabled in launcher settings for each version.
- Use short-lived sessions (already implemented) and validate signature on every startup.
